# ClaimScope — Data

This folder holds all data produced and consumed by the ClaimScope system — from the original Excel source file through to the final DuckDB analytical database. Everything here is generated by the pipeline and is **gitignored** (except this README). It must be created locally by running `python -m pipeline.run_pipeline`.

---

## Folder structure

```
data/
├── raw/
│   └── claim.xlsx                    ← Source data (place here before running pipeline)
│
├── processed/
│   ├── raw_claims.parquet            ← Step 1: Excel converted to parquet, no transformations
│   ├── validation_report.json        ← Step 2: Data quality report (nulls, ranges, duplicates)
│   ├── validation_summary.parquet    ← Step 2: Machine-readable quality summary
│   ├── claims_cleaned.parquet        ← Step 3: Normalised + flagged claims
│   ├── claims_featured.parquet       ← Step 4: Feature-engineered claims (peer z-scores, residuals)
│   └── claims_anomaly.parquet        ← Step 5: Final scored claims with ANOMALY_SCORE and ANOMALY_REASONS
│
├── marts/
│   ├── kpi_mart.parquet              ← Portfolio-level KPI aggregates
│   ├── warranty_mart.parquet         ← Per-warranty aggregates
│   ├── geography_mart.parquet        ← Per-region and per-province aggregates
│   ├── vehicle_mart.parquet          ← Per-brand and per-model aggregates
│   ├── trend_mart.parquet            ← Monthly time-series of claim volume and paid
│   ├── anomaly_mart.parquet          ← Anomaly-flagged claim records with reason strings
│   └── claim_detail_mart.parquet     ← Full per-claim records for the detail modal
│
└── duckdb/
    └── warrantywise.duckdb           ← Serving database (all marts loaded as tables)
```

## Raw data

### File requirements

| Property | Expected value |
|----------|---------------|
| Filename | `claim.xlsx` |
| Location | `data/raw/claim.xlsx` |
| Sheet name | `Claim` |
| Format | `.xlsx` (Excel 2007+) |

The file is **not committed** to the repository. Obtain it from the hackathon dataset and place it at the path above before running the pipeline.

### Key columns (expected in the source file)

| Column | Description |
|--------|-------------|
| `CLAIM_ID` | Unique claim identifier |
| `CLAIM_DATE` | Date the claim was filed |
| `WARRANTY_TYPE` | Type of warranty under which the claim falls |
| `CLAIM_AMOUNT_PAID` | Indemnity amount paid by the insurer (€) |
| `PREMIUM_AMOUNT_PAID` | Premium proxy input — used as imbalance denominator only, not true actuarial premium |
| `CLAIM_REGION` | Geographic region of the claim |
| `CLAIM_PROVINCE` | Province within the region |
| `VEHICLE_BRAND` | Manufacturer of the insured vehicle |
| `VEHICLE_MODEL` | Model of the insured vehicle |
| `AGE_BUCKET` | Policyholder age group |
| `GENDER` | Policyholder gender |

> ⚠️ **Important:** `PREMIUM_AMOUNT_PAID` is an imbalance proxy input, not an actuarial loss ratio denominator. No policy exposure data is available. All ratio metrics in the dashboard are labelled as "imbalance proxy" accordingly.

## Processed files

Each file is a Polars-compatible Apache Parquet file. They form a linear dependency chain — each step reads the previous step's output.

### `raw_claims.parquet`
Exact copy of the source Excel data in columnar format. No transformations applied. Used as the starting point for all downstream steps.

### `validation_report.json`
Human-readable data quality report. Contains:
- Null count and null rate per column
- Value range (min/max) for numeric and date columns
- Duplicate claim ID count
- Categorical cardinality per string column

Example:
```json
{
  "total_rows": 52000,
  "null_counts": { "CLAIM_AMOUNT_PAID": 0, "WARRANTY_TYPE": 12 },
  "duplicates": 0,
  "date_range": { "min": "2020-01-01", "max": "2024-12-31" }
}
```

### `validation_summary.parquet`
Machine-readable version of the validation report, consumed by the `/health` endpoint to show data readiness in the backend.

### `claims_cleaned.parquet`
Fully normalised and enriched claims. Adds:
- `HIGH_COST_FLAG` (bool) — `True` if `CLAIM_AMOUNT_PAID` ≥ P95
- `CLAIM_TO_PREMIUM_RATIO` (float) — imbalance proxy per claim
- Type-correct date columns
- Normalised string columns (trimmed, consistent casing)

### `claims_featured.parquet`
Adds engineered features for anomaly detection:
- `PEER_MEAN`, `PEER_STD` — group statistics by warranty × region
- `Z_SCORE` — how many standard deviations this claim's paid amount is from its peer group
- `EXPECTED_VALUE_RESIDUAL` — actual paid minus group expected value
- Normalised ratio and vehicle-age features

### `claims_anomaly.parquet`
Final enriched dataset. Adds:
- `ANOMALY_SCORE` (float 0–1) — combined IsolationForest + z-score signal
- `ANOMALY_FLAG` (bool) — `True` if score exceeds threshold
- `ANOMALY_REASONS` (list[str]) — human-readable strings explaining the flag

This is the primary input for all mart aggregations.

## Mart files

All parquets in `marts/` are pre-aggregated views of `claims_anomaly.parquet`. They are loaded into DuckDB as tables by `pipeline/marts.py`.

| Mart | Primary key | Key metrics |
|------|-------------|-------------|
| `kpi_mart` | Portfolio (single row) | total_claims, total_amount_paid, avg_claim_paid, p95_claim_paid, anomaly_rate, avg_claim_to_premium_ratio |
| `warranty_mart` | warranty_type | claim_count, total_claim_paid, avg_claim_paid, anomaly_rate, avg_claim_to_premium_ratio |
| `geography_mart` | region / province | claim_count, avg_claim_paid, high_cost_share, avg_claim_to_premium_ratio |
| `vehicle_mart` | brand / model | claim_count, total_claim_paid, avg_claim_paid, concentration_share |
| `trend_mart` | period (YYYY-MM) | claim_count, total_claim_paid per month |
| `anomaly_mart` | claim_id | anomaly_score, anomaly_flag, anomaly_reasons (all flagged claims) |
| `claim_detail_mart` | claim_id | All columns — used for the Claims page and detail modal |

## DuckDB database

`duckdb/warrantywise.duckdb` is the single-file OLAP database that the backend API reads from at runtime. It contains all seven mart tables as registered views over the parquet files.

### Connecting directly (for debugging)

```python
import duckdb
con = duckdb.connect("data/duckdb/warrantywise.duckdb", read_only=True)
con.execute("SHOW TABLES").fetchall()
con.execute("SELECT COUNT(*) FROM warranty_mart").fetchone()
```

### Regenerating the database

```bash
python -m pipeline.run_pipeline
```

This is safe to re-run. All parquet files and the DuckDB file are overwritten in place.

> ⚠️ Stop the backend API before re-running the pipeline. DuckDB does not support concurrent write access from multiple processes.

## Storage notes

- `data/raw/` — small (single Excel file, ~5–20 MB typical)
- `data/processed/` — grows linearly with row count; ~4 parquet files × dataset size
- `data/marts/` — small; pre-aggregated summaries
- `data/duckdb/` — typically under 50 MB for a claims dataset of ~50K rows

All contents of `data/` are gitignored. Do not commit claim data to the repository.
